The Project

About FitTracker

FitTracker is a fully functional fitness analytics web application developed for ISM 6225 — Application Development for Analytics at the University of South Florida. The platform enables users to log workout sessions using full CRUD operations backed by localStorage, visualize progress through interactive Chart.js dashboards, browse exercises fetched live from an external REST API, and receive personalized guidance through an AI-powered chatbot.

The application follows a JavaScript MVC architecture: Models (localStorage via storage.js), Views (HTML pages), and Controllers (workoutController.js and apiController.js). All data persists across browser sessions using the Web Storage API.

API Integration

External API: wger REST API

The Exercise Library page fetches live data from the wger REST API — a free, open-source fitness API with no authentication required.

Base URL:
https://wger.de/api/v2/
Endpoints Used:
GET /exercisecategory/?format=json — Fetches all muscle group categories
GET /exerciseinfo/?format=json&language=2&limit=20 — Fetches exercises with English names, muscles, and equipment
Implementation:
Calls are made using the browser's native fetch() API in js/apiController.js. Responses are normalized and rendered as cards on exercises.html. Results support category filtering and text search. Any exercise can be added to the workout log via URL parameters passed to create.html.

Architecture

JavaScript MVC Pattern

MODEL

js/storage.js manages all data. The FitStorage singleton reads and writes to localStorage, handles seed data initialization, and provides computed stats and chart-ready weekly/category breakdowns.

VIEW

Each HTML page is a View. Pages are responsible only for rendering data and capturing user input. They delegate data logic to the controllers and never access localStorage directly.

CONTROLLER

js/workoutController.js handles CRUD business logic. js/apiController.js manages external API calls. Controllers are the only bridge between Views and the Model.

Data Architecture

Logical Data Model (ERD)

Three-entity normalized structure: Users → Workouts and Workouts → Exercises (one-to-many relationships).
